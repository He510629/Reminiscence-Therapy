# 静态演示版 GitHub Pages 设计

## 目标

把当前项目整理成一个“打开链接即可浏览”的静态演示版，方便发布到 GitHub Pages 给其他人直接查看，不再依赖本地后端、数据库和登录流程。

## 适用范围

本次目标是“看页面、可基本交互地浏览演示内容”，不是完整上线可登录产品。因此设计重点是：

- 去掉登录门槛
- 让前端脱离本地 FastAPI 运行
- 保留现有主要页面和视觉样式
- 用前端内置演示数据支撑页面展示

## 推荐方案

采用“前端静态演示版”方案：

1. 应用打开后直接进入策展馆首页，不再进入登录页。
2. 认证上下文改成演示模式，始终提供一个默认演示用户。
3. 原本通过接口获取的数据，改为从前端本地 mock 数据读取。
4. 保留页面路由、详情跳转和游戏演示流程，让别人点开链接后仍然像在浏览一个完整产品。
5. 为 GitHub Pages 适配路由基础路径和静态资源路径。

## 页面体验

### 首页入口

- 根路径直接进入 `/museum`
- 不显示登录、注册入口
- 顶部保留“认知陪伴”品牌与当前用户区域，但退出按钮需要改成“演示版”说明或直接隐藏

### 策展馆

- 完整保留目前已做好的重点内容
- 继续支持推荐页、全部内容页、详情页跳转
- 使用本地演示数据保证内容稳定可见

### 健脑营

- 保留列表和题目演示
- 游戏结果、历史记录等如需后端写入的能力改为前端本地模拟

### 陪伴室与报告

- 若页面当前强依赖后端返回，可改为展示一组固定演示内容
- 原则是不让用户点击后出现空白或报错

## 数据方案

新增前端演示数据层，作为静态演示版统一数据来源：

- 演示用户
- 策展内容列表与详情
- 游戏列表、题目与结果
- 报告与聊天示例

实现上优先采用单独的 `demo` 数据目录，而不是把 mock 数据散落在页面组件里。这样后续继续补内容时更容易维护。

## 认证方案

保留现有 `AuthProvider` 外壳，但切换到演示模式：

- 默认 `isLoggedIn = true`
- 默认注入一个演示用户对象
- `logout()` 不再真正退出，可替换成无操作或跳回首页

这样可以最大程度复用现有受保护路由结构，减少对页面层的大改动。

## API 方案

现有 `frontend/src/api/index.ts` 直接请求 `/api/v1/*`，不适合 GitHub Pages。

推荐改法：

1. 保留现有 API 方法名，避免页面层大面积改动。
2. 在 API 层新增“演示模式”分支，方法内部直接返回本地 Promise 数据。
3. 让页面代码继续调用 `contentApi`、`gameApi`、`reportApi`、`chatApi`，但数据来自本地 mock。

这样改动集中，后续如果要切回真实接口也更容易。

## 路由与部署方案

GitHub Pages 对前端路由有两个关键要求：

1. 需要配置 `basename`
2. 需要处理刷新 404 问题

推荐做法：

- 使用 `BrowserRouter` + `basename`
- 补充 GitHub Pages 常见的 `404.html` 回退方案，保证深链接可用

如果实现复杂度超出当前项目结构，可退一步改为 `HashRouter`，这样部署最稳，但 URL 会带 `#`。优先尝试 `basename + 404`，如验证不稳再切 `HashRouter`。

## 发布结果

最终产物应满足：

- 打开公开链接后直接进入页面
- 不需要登录
- 可浏览策展内容和主要演示页面
- 不会因为缺少后端而报错
- 可以推送到 GitHub 并发布到 GitHub Pages

## 风险与边界

- 演示版不提供真实登录、注册、数据保存能力
- 聊天、报告、游戏成绩等属于演示数据，不会持久化
- 如果某些页面当前对后端耦合过深，可以优先保证页面可看，再逐步补全演示交互

## 受影响模块

- `frontend/src/App.tsx`
- `frontend/src/contexts/AuthContext.tsx`
- `frontend/src/api/index.ts`
- 各主要页面组件
- Vite 构建配置与 GitHub Pages 发布相关文件

## 成功标准

1. 本地执行前端构建成功
2. 构建产物在静态服务器中可直接访问
3. 根路径打开后直接进入页面
4. 策展馆、详情页、健脑营等主要入口可访问
5. 推送到 GitHub Pages 后他人可直接打开链接查看
