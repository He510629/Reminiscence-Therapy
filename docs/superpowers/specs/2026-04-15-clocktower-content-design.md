# 钟楼旧影内容改造设计

## 目标

将现有“钟楼旧影”从一条简短占位内容，升级为一条适合老年用户阅读的怀旧策展内容。内容形式保持为单条展品，但正文改为按年代串讲的故事型叙述，让老人读起来像在听人慢慢讲西安钟楼这些年的变化与记忆。

## 用户体验目标

- 保留“钟楼旧影”这一条内容，不拆成多条卡片。
- 增加钟楼相关图片，尽量覆盖不同年代。
- 文案不走资料罗列风格，而是采用温和、人性化、可共情的讲述方式。
- 每个年代段落都包含“大事、趣事、生活场景、老人容易代入的细节”。

## 现状约束

- 后端 `Content` 数据结构当前只有单个 `media_url` 和 `thumbnail_url` 字段，没有多图数组。
- 前端策展馆列表页和详情页目前实际展示的是单个占位视觉块，并未读取多图内容。
- 因此若要实现“多年代图片 + 长故事正文”，需要在本轮中做最小扩展，而不是只改数据库种子文案。

## 方案选择

采用“单条内容 + 多年代故事正文 + 详情页多图展示”的方案。

原因：

- 符合用户希望的“像听故事一样”的阅读体验。
- 比拆成多卡片更连贯，情绪更完整。
- 改动范围可控，只需要增强单条内容详情，不必重做策展馆整体结构。

## 数据设计

对“钟楼旧影”这条内容增加一份结构化扩展信息，优先复用现有 `theme_tags` 的 JSON 能力或在描述字段中存放结构化片段并在前端解析；如果现有方式不够清晰，则对内容模型补充一个最小字段用于保存 `gallery` / `story_sections`。

推荐的数据内容：

- `cover_image`: 钟楼主图
- `gallery`: 3 到 4 张不同年代钟楼图片
- `story_sections`: 按年代组织的故事段落，例如 `1960s`、`1970s`、`1980s`、`今天再看`

每个故事段落包含：

- 年代标题
- 对应图片
- 叙述正文
- 可选的小标题，如“大事记”“街坊趣闻”“那时的日常”

## 内容策略

图片来源策略：

- 优先寻找公开可引用的钟楼历史照片、地方志资料图、博物馆或档案馆公开页面。
- 如果某个年代找不到严格对应年份的公开图，则选取最接近的年代图并在内容整理时保持时间表述谨慎。

文案策略：

- 用第二人称和陪伴式表达，减少生硬说明。
- 内容多写生活感，如逛街、过节、听广播、坐公交、看城里热闹、钟楼附近的人声与烟火气。
- 每段都让老人能够“想起自己那时候是怎么过日子的”，而不是只知道钟楼发生了什么。

## 前端展示设计

列表页：

- 仍显示“钟楼旧影”这张内容卡。
- 若条件允许，列表卡面改为真实钟楼封面图。

详情页：

- 顶部显示封面图。
- 下方增加按年代排列的图文段落。
- 保留朗读按钮，但朗读内容优先读取故事正文。

## 后端与内容更新

- 更新“钟楼旧影”这条初始内容的数据。
- 保证老数据库中的现有记录也能被同步更新，而不是只修改种子逻辑。
- 若需要图片静态资源落地到本地项目，统一放在后端可访问的上传或静态目录下。

## 验证方式

- 打开策展馆列表，确认“钟楼旧影”显示真实钟楼图。
- 打开详情页，确认可看到多年代图文内容。
- 验证前端页面可正常加载图片。
- 验证文字长度下页面仍可正常滚动阅读。

## 风险与处理

- 历史图片版权和可用性不稳定：优先采用明确公开可访问来源，并保留来源记录。
- 现有数据结构不支持多图：采用最小字段扩展，不做大规模内容系统重构。
- 本地已有旧数据库记录：需要补一段同步逻辑或直接更新数据库记录。
